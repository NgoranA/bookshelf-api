# 06 — Deployment (ship it)

The last leg of the golden thread: *requirements → design → build → secure → test → document →
**ship**.* For the live session you don't have to deploy in real time (accounts + DNS eat the
clock) — **walking through these files is enough**. But everything here works end-to-end if you
want to do it for real.

We deploy with **no Docker** — students are new, and one new platform tool beats two. Render
runs our app on its **native Node runtime**: it installs Node, runs our build, and starts the
server, exactly like we do locally.

Three things every deployment of this service needs:

1. **The code + a Node runtime** — the platform installs Node and runs `npm ci`.
2. **A PostgreSQL database** — a *managed* one in the cloud (you don't run your own).
3. **Configuration injected as environment variables** — `DATABASE_URL`, `JWT_SECRET`,
   `NODE_ENV` — never committed to git.

---

## Step 1 — Prove it runs in "production mode" locally

Before any cloud, confirm the app runs with **no dev tooling** — no `--watch`, no
`pino-pretty`, production deps only. This is exactly what the platform will do.

```bash
cd bookshelf-api
npm ci                      # clean install from package-lock (what CI/the platform uses)

NODE_ENV=production \
DATABASE_URL='postgres://postgres:postgres@localhost:5432/bookshelf' \
JWT_SECRET='a-long-random-secret' \
node src/server.js          # no --env-file, no --watch — just the server

curl http://localhost:3000/health     # → {"status":"ok"}
```

🗣️ Two things to notice. We pass config **inline as environment variables**, not from a
`.env` file — *in production the platform injects configuration; the code contains no secrets.*
And with `NODE_ENV=production`, logs come out as **JSON** (not pretty) and error `detail` is
**hidden** from clients — both correct for a real deployment.

## Step 2 — Deploy to the cloud (Render, the simple path)

[Render](https://render.com) gives a free web service **and** a free managed PostgreSQL, and it
understands the `render.yaml` **Blueprint** shipped in `bookshelf-api/` — no Dockerfile needed.

**What `render.yaml` says** (this is the real file):
```yaml
services:
  - type: web
    name: bookshelf-api
    runtime: node                     # Render installs Node — no Docker
    plan: free
    buildCommand: npm ci              # install exactly what's in package-lock
    startCommand: node src/server.js  # how to run it
    healthCheckPath: /health          # Render polls this to know we're alive
    preDeployCommand: node scripts/migrate.js   # create tables before going live (safe)
    envVars:
      - key: NODE_ENV
        value: production
      - key: JWT_SECRET
        generateValue: true           # Render generates a strong random secret for us
      - key: DATABASE_URL
        fromDatabase:                 # Render injects the DB's connection string
          name: bookshelf-db
          property: connectionString
databases:
  - name: bookshelf-db
    plan: free
```

**Click-by-click:**

1. Push this repo to GitHub.
2. On Render: **New → Blueprint**, point it at your repo. Render reads `render.yaml`, creates
   the database and the web service, generates `JWT_SECRET`, and wires `DATABASE_URL`
   automatically.
3. First deploy runs `buildCommand` (`npm ci`), then `preDeployCommand`
   (`node scripts/migrate.js`) → tables are created.
4. *(Optional)* Seed once: open the service's **Shell** and run `node scripts/seed.js` (this
   adds the `demo@bookshelf.test` / `password123` account and sample books).
5. Visit `https://<your-service>.onrender.com/health` and `/docs`. On a live deploy you don't
   paste the secret anywhere — instead **register** an account (or log in as the seeded demo
   user), copy the returned **token**, and paste *that* into **Authorize**.

🗣️ Four things to point out:
- **No Docker, no build step of our own.** `runtime: node` + `buildCommand: npm ci` is the
  whole "build." The platform owns the Node version and OS.
- **Migrations vs reset.** `scripts/migrate.js` uses `CREATE TABLE IF NOT EXISTS` — safe to run
  on every deploy, never drops data. `db:reset` (which DROPs) must *never* point at prod.
- **Health checks.** `/health` is how the platform (and load balancers) know an instance is
  ready for traffic.
- **Secrets.** The DB password lives only in Render's injected `DATABASE_URL`; `JWT_SECRET` is
  generated by Render and signs login tokens. Neither is ever in git.

## Other good options (same shape, different button)

All native-Node, no Docker:

- **Railway** — connect the repo, add a PostgreSQL plugin, set `DATABASE_URL` from it; build
  `npm ci`, start `node src/server.js`, set `JWT_SECRET` and `NODE_ENV=production`.
- **Fly.io** — `fly launch` can build a Node app without a Dockerfile (Nixpacks);
  `fly postgres create` + `fly postgres attach` sets `DATABASE_URL`; set secrets with
  `fly secrets set JWT_SECRET=...`; run migrations via `fly ssh console -C "node scripts/migrate.js"`.
- **A plain VM (EC2/Droplet) + managed DB** — most control, most work: install Node, run the
  app under a process manager (e.g. `systemd` or `pm2`), put Nginx in front, manage TLS yourself.

The trade-off (from the cloud chapter of the notes): **PaaS** (Render/Railway) = least ops,
fastest; **a VM** = most control, most to maintain; **serverless** = great for spiky/low
traffic, watch out for cold starts and per-request DB connections. For a junior's first
service, **PaaS is the right call.**

## Production checklist

Before calling a service "shipped":

- [ ] `DATABASE_URL` and `JWT_SECRET` come from the environment, never committed (`.env` is gitignored).
- [ ] Migrations run on deploy (`scripts/migrate.js`), and `db:reset` is nowhere near prod.
- [ ] `NODE_ENV=production` (so error `detail` is hidden and logs are JSON).
- [ ] A health check endpoint exists and the platform is configured to use it (`/health`).
- [ ] Logs go to stdout as structured JSON (✅ pino, already wired).
- [ ] **helmet + CORS + rate limiting** are on (✅ already wired; set `CORS_ORIGIN` to your real
      frontend, not `*`, and consider a shared rate-limit store like Redis for multiple instances).
- [ ] Auth required on all book/review routes (✅ real accounts + JWT). Keep `JWT_SECRET` strong;
      rotating it invalidates every existing token, so plan for it.
- [ ] Tests pass in CI before anything deploys (see [`05-going-further.md`](05-going-further.md) → GitHub Actions).
